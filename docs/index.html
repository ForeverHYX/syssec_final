HardenInspector

一个面向 APK 的静态加固技术证据链检测器。它不判断应用是否恶意，而是回答： 这个 APK 静态上出现了哪些加壳、代码混淆、环境检测和 Native 入口证据。

这个项目到底做什么

把 APK 中散落的静态证据提取出来，组织成可解释的加固技术画像，供人工分析和课程展示使用。

输入是什么

输入是一个 Android APK 文件。HardenInspector 将 APK 当作 ZIP 容器读取，识别 Manifest、DEX、Native `.so`、assets 和资源文件。

APK ZIP AndroidManifest.xml classes*.dex lib/**/*.so

输出是什么

输出是 JSON 或文本报告。每条 finding 都有规则 ID、类别、置信度、解释和 evidence，能回到具体文件位置和值。

schema_version summary findings evidence

它不做什么

它不是恶意软件判定器，不承诺商业壳全覆盖，也不做动态 Frida 验证。它给出的是静态可观测证据和人工复核入口。

非恶意判定 非完整反编译器 非商业签名库
展示时的一句话： HardenInspector 的核心价值是“静态 evidence chain”：每个检测结论都能追溯到 APK 内的 Manifest 字符串、DEX 字符串/方法/opcode、Native ELF 符号或资源熵值，而不是只给一个黑盒风险分。

总体框架设计

框架遵循中期报告的解析层、特征层、决策层和报告层。期末交付保留静态核心链路，把动态验证层明确放在当前范围之外。

1. APK 输入 接收本地 APK 路径、Web demo 预置样本或浏览器上传 APK bytes。
2. 容器解析 读取 ZIP entry、文件大小、压缩大小、SHA-256、Shannon 熵。
3. 多源特征 提取 Manifest string pool、DEX 字符串/方法/opcode、Native strings/ELF symbols。
4. 特征模型 归一化为 `ApkFeatures`，让规则层不用关心底层文件格式细节。
5. 规则引擎 执行解释型静态规则，生成带类别、置信度和证据的 finding。
6. 展示与评估 输出 CLI JSON/text、Web demo、数据集报告和 benchmark 指标。

为什么不用完整逆向框架做运行时依赖

课程展示强调可复现和离线可运行。项目只需要检测相关的少量证据，不需要完整反编译、控制流恢复或 UI 反编译。因此默认运行时使用 Python 标准库和轻量解析器，Androguard 作为 benchmark 基线，不作为核心依赖。

为什么强调 evidence chain

加固、插件化、调试检测和正常业务逻辑之间存在重叠。只输出标签很容易误导。HardenInspector 把 `location` 和 `value` 放进报告，让展示者能解释“为什么命中”和“哪些证据需要人工复核”。

核心模块分工

模块 职责 展示时怎么讲
src/hardeninspector/apk.py 打开 APK ZIP，识别 Manifest、DEX、Native 库、资源和文件熵。 这是入口层，先把 APK 拆成可分析对象。
src/hardeninspector/axml.py 提取 Android binary XML string pool 中的 Manifest 字符串。 用来看到壳入口类、权限、包名和 Manifest 层常量。
src/hardeninspector/dex.py 解析 DEX strings、types、methods、class data、code item、`const-string`、`invoke-*` 和 opcode profile。 这是 Java/Kotlin 层证据来源，覆盖反射、动态加载、短类名和控制流密度。
src/hardeninspector/native.py 扫描 Native strings，解析 ELF `.dynsym` / `.symtab` 符号。 说明工具不只看 DEX，也能看到 `JNI_OnLoad`、`ptrace`、`dlopen` 等 Native 证据。
src/hardeninspector/features.py 将多源解析结果归一化为 `ApkFeatures` 和 `StringEvidence`。 这是解析层与规则层的边界，降低规则实现耦合。
src/hardeninspector/rules.py 执行 packer、obfuscation、environment、native 四类规则，生成 `Finding`。 规则是解释型的，每条都有证据，不是机器学习黑盒。
src/hardeninspector/report.py 串联 `extract_features` 和 `evaluate_rules`，输出 `DetectorReport`。 CLI、Web demo、数据集报告共用同一个扫描管线。
src/hardeninspector/demo_web.py 用 Python 标准库启动本地 HTTP demo，支持预置样本、上传扫描和 benchmark 展示。 现场演示入口，证明工具不是只生成离线报告。

报告模型与证据链

扫描结果围绕 `DetectorReport` 组织：APK 元数据、四类 summary、finding 列表和 evidence 列表。

{
  "schema_version": 1,
  "apk": {
    "path": "combined_hardened_showcase.apk",
    "sha256": "...",
    "entry_count": 7
  },
  "summary": {
    "packer": 4,
    "obfuscation": 2,
    "environment": 3,
    "native": 1
  },
  "findings": [
    {
      "id": "environment.system_properties",
      "category": "environment",
      "confidence": "high",
      "evidence": [
        {
          "kind": "dex-string",
          "value": "ro.kernel.qemu",
          "location": "classes.dex:string"
        }
      ]
    }
  ]
}

怎么解释一条 finding

  • id 是稳定规则名，便于文档、测试和 benchmark 对齐。
  • category 映射到课程目标中的加壳、混淆、环境检测和 Native 证据。
  • confidence 表示规则可信度，高熵 payload、动态加载、反射等可能需要结合上下文解释。
  • evidence.location 告诉你证据来自 Manifest、DEX、Native 符号表还是资源文件。
  • evidence.value 是可复查的原始值，例如壳库名、API 名、系统属性、文件路径或 ELF 符号。

检测规则覆盖

规则不是孤立字符串匹配，而是尽量结合位置、结构和上下文，降低“正常框架代码被误判为加固”的风险。

加壳 Packer

识别已知壳库名、Manifest StubApp、高熵 payload、动态加载 API 和 Native loader 符号。

libjiagu.so StubApp DexClassLoader dlopen

混淆 Obfuscation

识别短类名比例、反射 API / `Class.forName` / `Method.invoke` 和控制流 opcode 密度。

La/a; forName invoke-* goto / switch

环境检测 Environment

覆盖模拟器、调试器、ADB 设置、安装来源、签名完整性、root、Frida/Xposed 和 Native anti-debug。

ro.kernel.qemu ADB_ENABLED ptrace /proc/self/maps

Native 入口

识别 `JNI_OnLoad` 和 `Java_*` JNI 导出符号，把 Native 桥接作为独立证据类别展示。

JNI_OnLoad Java_* .dynsym .symtab

误报控制口径

风险点 为什么容易误报 当前处理
反射 Android support、AndroidX、Kotlin 和 Material 兼容层会有正常反射样板。 过滤常见库前缀，要求强字面量或应用自有上下文证据。
动态加载 插件化框架也会使用 `DexClassLoader` 或 `PathClassLoader`。 作为 medium confidence finding，建议结合壳库、高熵 payload、StubApp 一起解释。
高熵文件 压缩资源、模型文件、媒体文件也可能高熵。 只对 `assets/` 中达到大小和熵值阈值的文件输出 medium confidence 证据。
控制流密度 普通复杂业务逻辑也会有分支。 要求指令数、控制流 opcode 数和密度同时超过阈值，只作为轻量预筛。
root 探测 裸 `su` 子串可能出现在普通单词中。 只匹配强 root artifact、root 管理包名、`test-keys` 或 `which su` 等上下文。

数据集与指标怎么讲

评估由可控 oracle 样本和外部现成 APK 共同组成，既能计算 ground truth，也能观察真实 APK 复杂度。

22 个合成 oracle APK

每个样本都控制植入某类证据，例如 R8 短标识符、Obfuscapk 风格反射、Bangcle/Jiagu 风格壳、Native ptrace loader、ADB 设置、安装来源、签名完整性和综合样本。

可复现 有 labels.json 有 JSON reports

12 个外部现成 APK

包含 DroidBench、F-Droid 和 PIVAA。外部样本用于扩大解析覆盖、观察误报，并通过 `label_basis` 记录粗粒度类别映射依据。

答辩速查

这里放最可能被问到的高频问题。完整清单见 docs/defense_qa.md。

Q1：HardenInspector 和恶意软件检测有什么区别？

它不判断恶意或良性，只检测 APK 中可观测的加固、混淆、环境检测和 Native 入口证据。加固可能出现在安全防护、商业保护、插件化或恶意规避场景中，所以报告只给证据链，不直接下恶意结论。

Q2：为什么不用 APKiD 或 Androguard 直接完成？

课程目标是实现检测器。APKiD、Androguard 和 ZIP Strings 在本项目中是比较对象。HardenInspector 的贡献是把 Manifest、DEX、Native、资源熵值统一到一个 evidence-chain 模型里，并覆盖课程目标下的四类规则。

Q3：Micro F1 = 1.000 会不会过拟合？

这个分数只代表当前 34 个合并评分样本和四类粗标签定义下的结果，不代表全网 APK。防守点是：22 个合成 oracle APK 可精确计算 ground truth，12 个外部 APK 扩大解析覆盖，所有标签依据和报告都提交在仓库里可复查。

Q4：为什么合成 APK 有意义？

加固 ground truth 很难从真实 APK 中公开、合法、稳定地获取。合成 APK 的价值是可控植入证据，让每条规则有可计算 oracle。外部 APK 则补充真实复杂度和误报观察，两者解决的问题不同。

Q5：Native 检测做到了什么程度？

当前实现扫描 Native strings，并解析 ELF `.dynsym` / `.symtab`，能识别 `JNI_OnLoad`、`Java_*` JNI 导出、`ptrace`、`prctl`、`syscall`、`dlopen`、`android_dlopen_ext` 等符号。它不做 Native 反汇编和语义恢复。

Q6：如何证明 Web demo 不是写死结果？

Web demo 的预置样本和上传扫描都调用同一个 `scan_apk` pipeline。上传 APK 会写到本地临时目录，扫描后删除，并返回正常 JSON report。可以现场上传仓库中的任意 APK 复核。

Q7：最大的局限是什么？

不做完整 CFG、深度 Native 反汇编或动态 Frida 验证，不承诺商业壳签名完备覆盖。项目定位是课程级可复现静态检测框架，而不是工业级移动威胁情报平台。
